Crypto Demystified

Encoding, Hashing & Encryption

Everything you use every day but were afraid to ask about.

Encoding

Transforms data format. Anyone can reverse it. Not security.

Hashing

Fingerprints data. Nobody can reverse it. One-way.

Encryption

Locks data with a key. Only the key holder can reverse it.

Encoding

Hex Encoding

Represents every byte as two characters (0-9, a-f) — a human-readable view of raw binary.

Analogy
Spelling out a phone number digit by digit. "Five-five-five-one-two-three-four." Same info, just laid out character by character.
Try It
Type something:
48 65 6c 6c 6f
When to use

Reading binary in debuggers, log output, CSS color codes (#FF5733). Whenever you need to see raw bytes.

Reversible Deterministic No key needed
Encoding

Base64 Encoding

Converts binary data into text-safe characters (A-Z, a-z, 0-9, +, /) so it can travel through text-only channels.

Analogy
Writing a message using only the letters on a standard typewriter. The message isn't hidden — you just rephrased it so the typewriter can handle it.
Try It
Type something (watch the = padding):
SGVsbG8sIFdvcmxkIQ==
Common Mistake
Base64 is NOT encryption. Anyone can decode it instantly. When you see Authorization: Basic dXNlcjpwYXNz, that's just "user:pass" in Base64. It provides zero security.
When to use

Embedding images in HTML (data URIs), sending binary in JSON, email attachments, HTTP Basic Auth headers.

Reversible Deterministic No key needed
Encoding

URL Encoding

Replaces characters that have special meaning in URLs with %XX hex codes so they're treated as data, not syntax.

Analogy
Like escape characters in code. When you write \" inside a string, you're telling the parser "this quote is data, not syntax." Same idea for URLs.
Try It
Type a URL with special characters:
hello%20world%26foo%3Dbar
Common Mistake
Forgetting to encode & in a query value breaks parameter parsing. ?q=cats & dogs won't work — the & starts a new parameter. You need ?q=cats%20%26%20dogs.
Reversible Deterministic No key needed

Hashing

SHA-256

Takes any data and produces a fixed 64-character fingerprint. Change one letter and the entire fingerprint changes.

Analogy
A meat grinder. You put a steak in, you get ground beef out. You can verify a steak would produce that ground beef, but you can't reconstruct the steak.

Input B:

Change one character. Watch the entire hash change.

Common Mistake
SHA-256 alone is NOT safe for passwords. No salt means identical passwords produce identical hashes. Too fast — GPUs compute billions per second. Use bcrypt for passwords.
When to use

File integrity checks (checksums), git commit hashes, content-addressable storage, deduplication.

Not reversible Deterministic No key needed
Hashing

bcrypt

A deliberately slow hashing algorithm designed specifically for passwords, with a built-in random salt.

Analogy
A lock that takes 1 second to try each key. If you have one key, 1 second is fine. If an attacker is trying a billion keys, they'll be dead before they finish.
How It Works
Same password "hunter2" hashed twice — different output each time:
$2b$12$LJ3m4ys3Lk0TSwMCkVc8aOY2rGJsVU5X3MQz6...
$2b$12$9Q8fN7mP3xR2tY0wK5v1bOZJ8hG6dA4cE3nL1...
$2b = algorithm  |  $12 = cost factor (2^12 rounds)  |  blue = salt  |  green = hash
Common Mistake
Never store passwords as plain SHA-256 or MD5. bcrypt is intentionally slow (tunable), includes automatic salting so identical passwords get different hashes, and is purpose-built for this one job.
Not reversible Not deterministic (salt) No key (salt embedded)
Hashing

HMAC

A hash that requires a secret key — proves data hasn't been tampered with AND came from someone who knows the key.

Analogy
A wax seal on a letter. Everyone can read the letter. But the seal proves it came from the king and wasn't opened en route.

When to use

Webhook signature verification (Stripe, GitHub), JWT signing (HS256 = HMAC + SHA-256), API request signing (AWS Signature V4).

Not reversible Deterministic Requires secret key

Applied

JWT (JSON Web Token)

A token that bundles a JSON payload with a signature. Combines Base64 encoding and HMAC signing into one package.

Analogy
A transparent ID badge with a holographic seal. Everyone can read your name and role. The seal proves the badge was issued by your company and wasn't forged. But the info itself is right there for anyone to see.

Common Mistake
JWTs are signed, NOT encrypted. The payload is just Base64 — anyone can decode it and read the contents. Don't put secrets or sensitive PII in the payload.
Payload: reversible (Base64) Signature: not reversible Requires key for signing
Encryption

AES (Symmetric Encryption)

Same key encrypts and decrypts. Fast. The workhorse of actual encryption.

Analogy
A diary with a combination lock. You write in it and lock it. Anyone with the combination can read it. The challenge: how do you share the combination without someone overhearing?
When to use

Encrypting data at rest (database fields, disk encryption), encrypting data in transit (TLS uses AES), encrypting backups. Whenever you need actual secrecy and both sides share the key.

Key Point
The hard problem isn't the encryption — it's key management. You must securely store and distribute the key. If the key leaks, everything encrypted with it is compromised.
Reversible (with key) Not deterministic (IV) Same key encrypts + decrypts
Encryption

RSA (Public-Key Encryption)

Two keys: public encrypts, private decrypts. Solves the "how do I share a key securely" problem.

Analogy
A mailbox. Anyone can drop a letter through the slot (encrypt with public key). Only the owner with the mailbox key can open it and read the letters (decrypt with private key).
When to use

HTTPS/TLS handshake, SSH keys (ssh-keygen), JWT signing with RS256, code signing, PGP email.

Key Point
RSA is slow. In practice, it's used to exchange an AES key, then AES does the bulk encryption. This is how HTTPS works — RSA for the handshake, AES for the data. You almost never encrypt large data directly with RSA.
Reversible (with private key) Key pair (public + private) Slow — use for key exchange
